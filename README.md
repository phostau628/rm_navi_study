### 模块1：激光雷达数据接收与预处理
**代码对应函数**：`livox_pcl_cbk()`（Livox专用）、`standard_pcl_cbk()`（标准点云通用）
| 步骤序号 | 你的原始学习注释 | 核心知识点补充 |
|----------|------------------|----------------|
| 步骤1 | 创建两个点云容器（预处理完成后的点云 / 分割点云） | 两个容器分别对应：`ptr`(预处理输出的完整点云)、`ptr_div`(帧分割/合并用的临时点云) |
| 步骤2 | 对激光雷达数据进行预处理：<br>1. 解析雷达原始报文（Livox / 标准点云）<br>2. 计算每个点 x, y, z，3D坐标（关键）<br>公式：x = R(距离)* cos(elevation)垂直角 * cos(azimuth)水平角<br>y = R * cos(elevation) * sin(azimuth)<br>z = R * sin(elevation)<br>3. 计算每个点 intensity（强度）<br>物体越白 / 越亮 → intensity 越大 （用于：地面过滤，特征筛选）<br>4. 给每个点打上【微秒级时间戳】（存在 curvature 字段）<br>点的时间 = 帧起始时间 + 点的时间偏移（去畸变必须知道每个点是何时发射的）<br>5. 过滤太近 / 太远的点<br>6. 输出：一帧带时间戳的原始点云 | 这是LIO的基础前提，**每个点的独立时间戳是运动去畸变的核心**，Livox非重复扫描的特性必须依赖单点点时间戳 |
| 步骤3 | 如果cut_frame参数为true，按照时间戳对点云进行分割；如果con_frame参数为true，按照时间戳对点云进行合并；否则不进行分割和合并，直接把完整一帧放进队列 | 帧分割/合并是针对低帧率雷达/高动态场景的优化，默认关闭，直接完整帧入列缓存 |

| 核心环节 | 你的原始学习注释 | 核心知识点补充 |
|----------|------------------|----------------|
| 卡尔曼预测 | 第一步：卡尔曼预测，dt为当前IMU时间与上次预测的时间差<br>纯靠IMU，从上一时刻推算当前时刻的位姿<br>输入：dt(时间差), Q_output(过程噪声), input_in(IMU数据) | 预测环节只靠IMU高频积分，先给出一个粗略的位姿先验，为后续激光匹配提供初值 |

## 三、核心概念与疑问解答笔记（对应你之前的核心疑惑）
### 1. 坐标系相关疑问
| 疑问 | 结论 |
|------|------|
| 为什么代码里的“世界坐标系”叫`odom`，不叫`map`？ | 符合ROS REP-105官方约定：<br>- `odom`：里程计推算的相对世界，短时间精准，会有累积漂移，由里程计/LIO输出<br>- `map`：绝对固定的真实世界，需要回环检测/GPS修正漂移<br>这份代码是纯前端LIO，没有回环/后端优化，所以只用`odom`，不需要`map` |
| 为什么直接是`odom → lidar_link`，没有`base_link`中间层？ | 代码做了极简设计：直接把`lidar_link`当作机器人本体坐标系`base_link`；<br>雷达和IMU的外参只在代码内部用矩阵计算，不发布到TF树里，减少复杂度和计算开销 |
| 是不是只有publish了TF，才能被查到？ | 是的！TF是“广播-收听”系统，只有通过`tf_br->sendTransform()`发布到TF系统的变换，才能被`tf_echo`、RViz、其他节点查到；代码内部的矩阵乘法，不发布的话，外部完全查不到 |

### 2. 代码架构相关疑问
| 疑问 | 结论 |
|------|------|
| 算法核心到底在哪个文件？ | 核心位姿解算算法100%在`Estimator.cpp`里，包括卡尔曼预测/更新、点云配准、迭代优化；<br>你正在看的`laser_mapping.cpp`，只是ROS通信、数据调度、结果发布的“外壳”，不做核心算法计算 |
| `set_posestamp()`是算法函数吗？ | 不是！它只是一个赋值函数，把`Estimator`里已经算好的位姿，复制到ROS消息里，没有任何数学计算和位姿解算 |
